# Roll Support — Design Spec
**Datum:** 2026-06-27  
**Scope:** Fall 1 — Nackter Short Put (Cash Secured Put), Wheel-Strategie  
**Plattform:** Streamlit master, keine DB-Persistenz (Session-only)

---

## Ziel

Der User gibt seinen laufenden Short Put ein. Skuld zieht aktuelle Marktdaten aus der DB, berechnet den aktuellen Stand der Position und zeigt eine gefilterte Tabelle mit Roll-Kandidaten pro Stufe (1/2/3) — inklusive neuer Gewinnschwelle und Sinnvoll-Flag.

---

## Input-Felder (manuell)

| Feld | Typ | Beispiel |
|---|---|---|
| Symbol | Text | AAPL |
| Strike K | Float | 185.00 |
| Expiry | Datum | 2026-07-18 |
| Prämie eingenommen (P_eröffnung) | Float | 3.50 |
| Anzahl Kontrakte (n) | Int | 1 |
| Kumulierte Prämien bisheriger Rollen (P_kum) | Float | 0.00 |
| Aktuelle Roll-Stufe | Int (0/1/2/3) | 0 |

**Knopf:** "Analysieren" → triggert DB-Query + Berechnung

---

## Automatisch aus DB (OptionDataMerged)

- Aktueller Aktienkurs `S` → `live_stock_price` für Symbol
- `P_aktuell` → `day_close` für exakt diesen Strike + Expiry (Näherung Schließungskosten)
- Roll-Kandidaten Stufe 1: gleicher Strike K, alle Expiries mit DTE 25–70
- Roll-Kandidaten Stufe 2: Strikes < S (OTM), DTE 40–70
- Roll-Kandidaten Stufe 3: wie Stufe 2 (Kontrakte ×2, Kapitalcheck separat)

---

## Ausgabe Block 1 — Aktuelle Position

| Kennzahl | Formel |
|---|---|
| Gewinn/Verlust % | `(P_eröffnung - P_aktuell) / P_eröffnung` |
| P&L absolut | `(P_eröffnung - P_aktuell) × n × 100` |
| Innerer Wert | `max(0, K - S)` |
| Restzeitwert | `P_aktuell - Innerer_Wert` |
| Gewinnschwelle | `K - P_kum` |

---

## Ausgabe Block 2 — Roll-Kandidaten (3 Tabellen)

### Stufe 1 — Gleicher Strike, neue Laufzeit (30–60 DTE)

| Spalte | Quelle |
|---|---|
| Expiry | DB |
| DTE | DB |
| Prämie (P_roll) | DB · day_close |
| Netto | `P_roll - P_aktuell` |
| Neue Gewinnschwelle | `K - (P_kum + Netto)` |
| Sinnvoll? | Netto > 0 AND neue GS < alte GS |

### Stufe 2 — Niedrigerer Strike (OTM), 45–60 DTE

Wie Stufe 1, zusätzliche Spalte:

| Spalte | Quelle |
|---|---|
| Strike K2 | DB (alle OTM-Strikes unter S) |
| Neue GS | `K2 - (P_kum + Netto)` |
| Sinnvoll? | Netto > 0 AND neue GS < alte GS |

### Stufe 3 — Wie Stufe 2, aber 2 Kontrakte

Zusätzliche Spalten:

| Spalte | Formel |
|---|---|
| Netto (×2) | `(P_roll × 2 - P_aktuell × n) × 100` absolut |
| Netto pro Aktie | `Netto_absolut / (2 × 100)` |
| Kapital nötig | `K2 × 2 × 100` |
| Sinnvoll? | Netto > 0 AND neue GS < alte GS AND Kapital vorhanden (User-Check) |

## Datei-Struktur

```
pages/roll_support.py              ← Streamlit-Seite (Input + Ausgabe)
src/roll_support_calc.py           ← Berechnungslogik (pure Python, kein DB-Aufruf)
db/SQL/query/roll_candidates.sql   ← Query gegen OptionDataMerged
```

---

## Entscheidungsregeln "Sinnvoll?"

| Bedingung | Flag |
|---|---|
| Netto > 0 AND neue GS verbessert | ✅ |
| Netto > 0 aber GS nicht verbessert | ⚠️ |
| Netto ≤ 0 | ❌ |

---

## Nicht in Scope (V1)

- Bull Put Spreads (Fall 2) — separates Feature
- DB-Persistenz / Trade-Journaling — Ansatz B, späteres Feature
- Covered Call Phase (Endspiel) — späteres Feature
- Maximaler Verlust-Schwellwert / Stop-Loss-Regel — noch nicht definiert

---

## Offene Punkte

1. `P_aktuell` ist eine Näherung via `day_close` — kein echter Bid/Ask. Für exakte Schließungskosten muss User den Broker-Preis selbst prüfen.
2. Stufe 3 Kapitalcheck: Skuld zeigt "Kapital nötig", aber der User entscheidet ob er es hat.
3. Maximaler Verlust-Threshold: wann gibt man auf statt endlos zu rollen? Noch nicht definiert.
